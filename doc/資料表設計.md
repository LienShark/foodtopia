# Foodtopia MVP 資料表設計

本文整理 Foodtopia 網頁版 MVP 的資料表名稱、用途、主要欄位與關聯。

目前產品方向：

- 前端：Vue
- 後端：Python + FastAPI
- 第一階段：網頁版，不做 App
- 不做金流
- 不做 POS 串接
- 商品以「福袋」為主
- 商家由業務給審核碼，商家註冊時選擇 Google Maps 店家並輸入審核碼

## 角色定義

系統中先分成三種角色：

| 角色 | 說明 |
|---|---|
| Foodtopia 內部人員 | 業務、管理員、客服。MVP 可以先不做登入後台。 |
| 商家 | 早餐店、麵包店、餐廳、飲料店等合作店家。 |
| 一般使用者 | 從 LINE 或網頁進來，瀏覽福袋並預約取貨的人。 |

## 資料表總覽

目前規劃的資料表分成「MVP 先做」與「第二階段補上」。

### MVP 先做的核心資料表

第一版建議先做 4 張核心表：

| 表名 | 中文名稱 | 用途 |
|---|---|---|
| `merchants` | 店家 | 存一間實體店的資料，例如店名、地址、Google Maps ID、審核狀態。 |
| `merchant_users` | 商家使用者 | 存可以登入並管理某間店的人，例如老闆、店長、員工。 |
| `surprise_bags` | 福袋商品 | 存商家上架的福袋，例如今日早餐福袋、麵包福袋。 |
| `reservations` | 預約紀錄 | 存一般使用者預約福袋與取貨狀態。 |

### 第二階段補上的資料表

以下 2 張表也會做，只是可以比核心交易流程晚一點：

| 表名 | 中文名稱 | 用途 |
|---|---|---|
| `staff_users` | Foodtopia 內部人員 | 存業務、管理員、客服等內部帳號。 |
| `customers` | 一般使用者 | 存一般使用者資料，例如 LINE user ID、姓名、手機、預約紀錄關聯。 |

## staff_users：Foodtopia 內部人員

`staff_users` 代表 Foodtopia 團隊內部的人，不是店家。

例如：

- 業務
- 管理員
- 客服
- 營運人員

MVP 可以先不做完整內部後台，但這張表未來會用來管理商家審核、查看平台資料、處理客訴或管理審核碼。

### 欄位

| 欄位 | 說明 | 範例 | 是否必填 |
|---|---|---|---|
| `id` | 內部人員 ID | `staff_001` | 是 |
| `name` | 姓名 | `林業務` | 是 |
| `email` | 登入 email | `sales@example.com` | 是 |
| `password_hash` | 密碼雜湊，不存明文密碼 | `$2b$...` | 視登入方式 |
| `role` | 內部角色 | `admin`、`sales`、`support` | 是 |
| `status` | 帳號狀態 | `active`、`disabled` | 是 |
| `created_at` | 建立時間 | `2026-07-02T10:00:00+08:00` | 是 |
| `last_login_at` | 最後登入時間 | `2026-07-02T12:00:00+08:00` | 否 |

### role

| 角色 | 說明 |
|---|---|
| `admin` | 管理員，可以管理平台設定、商家與內部人員。 |
| `sales` | 業務，負責接洽店家、給審核碼、協助商家上線。 |
| `support` | 客服或營運，負責查看預約、處理問題。 |

### status

| 狀態 | 說明 |
|---|---|
| `active` | 可登入並使用內部後台。 |
| `disabled` | 已停用，不能登入。 |

### 和其他表的關係

第一版如果不做 `approval_codes` 表，`staff_users` 可以先不直接關聯其他資料表。

之後如果補上 `approval_codes`，可以設計成：

```text
staff_users 1 - N approval_codes
```

也就是一位業務可以發出多組審核碼。

## merchants：店家

`merchants` 代表「一間實體店」。

它不是登入帳號，而是店本身。例如：

- 晨光早餐店
- 巷口麵包坊
- 小島茶飲
- 立豐青菓行

### 欄位

| 欄位 | 說明 | 範例 | 是否必填 |
|---|---|---|---|
| `id` | 店家 ID，系統內部使用 | `merchant_001` | 是 |
| `name` | 店名 | `晨光早餐店` | 是 |
| `google_place_id` | Google Maps 店家的唯一 ID | `ChIJxxxxxx` | 是 |
| `address` | 店家地址 | `新竹市東區大學路100號` | 是 |
| `lat` | 緯度 | `24.786123` | 是 |
| `lng` | 經度 | `120.999456` | 是 |
| `phone` | 店家電話 | `03-1234567` | 否 |
| `business_type` | 店家類型 | `breakfast`、`bakery`、`restaurant`、`drink` | 是 |
| `status` | 店家狀態 | `approved`、`suspended` | 是 |
| `created_at` | 建立時間 | `2026-07-02T10:00:00+08:00` | 是 |
| `updated_at` | 更新時間 | `2026-07-02T10:00:00+08:00` | 是 |

### google_place_id 是什麼

`google_place_id` 是 Google Maps 對每一個地點提供的唯一識別碼。

存它的原因：

- 避免同一家店重複註冊。
- 可以從 Google Maps 帶入店名、地址、電話、經緯度。
- 之後可以做距離排序、地圖顯示、導航。
- 店名可能重複，但 `google_place_id` 比店名可靠。

### status 是什麼

`status` 控制這間店現在能不能在平台上正常運作。

| 狀態 | 說明 |
|---|---|
| `approved` | 已通過審核，可以上架福袋，也可以被使用者看到。 |
| `suspended` | 被暫停合作或停權，例如資料有問題、客訴、暫時不合作。 |

如果 MVP 的流程是「輸入正確審核碼就完成審核」，那商家註冊成功後可以直接變成 `approved`。

## merchant_users：商家使用者

`merchant_users` 代表「可以登入管理某間店的人」。

它和 `merchants` 的差別：

| 表 | 代表 |
|---|---|
| `merchants` | 一間店 |
| `merchant_users` | 管理這間店的人 |

一間店可能只有一個老闆帳號，也可能有多個員工一起管理。

### 欄位

| 欄位 | 說明 | 範例 | 是否必填 |
|---|---|---|---|
| `id` | 商家使用者 ID | `merchant_user_001` | 是 |
| `merchant_id` | 所屬店家 ID，對應 `merchants.id` | `merchant_001` | 是 |
| `name` | 使用者姓名 | `王老闆` | 是 |
| `phone` | 手機號碼 | `0912345678` | 否 |
| `email` | 登入 email | `owner@example.com` | 否 |
| `password_hash` | 密碼雜湊，不存明文密碼 | `$2b$...` | 視登入方式 |
| `line_user_id` | LINE 使用者 ID，未來接 LIFF 登入時使用 | `Uxxxxxxxx` | 否 |
| `role` | 權限角色 | `owner`、`staff` | 是 |
| `created_at` | 建立時間 | `2026-07-02T10:00:00+08:00` | 是 |
| `last_login_at` | 最後登入時間 | `2026-07-02T12:00:00+08:00` | 否 |

### role

| 角色 | 說明 |
|---|---|
| `owner` | 店主或主要管理者，可以修改店家資料、管理員工。 |
| `staff` | 店員，可以上架福袋、查看預約、完成取貨。 |

MVP 可以先全部都當 `owner`，之後再細分權限。

## surprise_bags：福袋商品

`surprise_bags` 代表商家上架的一個福袋。

你們目前不是一般電商，所以不需要一開始做很複雜的 `products`。第一版直接把可預約的東西稱為 `surprise_bags` 會更準。

例如：

- 今日早餐福袋
- 麵包驚喜包
- 熟食福袋
- 蔬果惜食包

### 欄位

| 欄位 | 說明 | 範例 | 是否必填 |
|---|---|---|---|
| `id` | 福袋 ID | `bag_001` | 是 |
| `merchant_id` | 哪間店上架的，對應 `merchants.id` | `merchant_001` | 是 |
| `title` | 福袋名稱 | `今日早餐福袋` | 是 |
| `category` | 福袋類型 | `bakery`、`meal`、`drink`、`produce`、`grocery`、`mixed` | 是 |
| `description_mode` | 內容揭露程度 | `hidden`、`partial`、`full` | 是 |
| `description` | 商家願意公開的內容 | `可能含三明治、飯糰、吐司` | 否 |
| `price` | 售價，單位為新台幣元 | `79` | 是 |
| `original_value` | 原價估值，用來呈現折扣感 | `150` | 否 |
| `quantity_total` | 總上架數量 | `10` | 是 |
| `quantity_reserved` | 已被預約數量 | `3` | 是 |
| `pickup_start_at` | 取貨開始時間 | `2026-07-02T18:00:00+08:00` | 是 |
| `pickup_end_at` | 取貨結束時間 | `2026-07-02T20:00:00+08:00` | 是 |
| `status` | 福袋狀態 | `draft`、`active`、`sold_out`、`expired`、`hidden` | 是 |
| `allergens` | 過敏原清單 | `["milk", "peanut"]` | 否 |
| `dietary_flags` | 飲食限制標籤 | `["contains_pork", "contains_beef"]` | 否 |
| `storage_note` | 保存或食用提醒 | `eat_today`、`refrigerated`、`frozen` | 否 |
| `created_at` | 建立時間 | `2026-07-02T10:00:00+08:00` | 是 |
| `updated_at` | 更新時間 | `2026-07-02T10:00:00+08:00` | 是 |

### description_mode

因為福袋內容要不要公開由商家決定，所以用 `description_mode` 控制。

| 模式 | 說明 | 使用者看到的感覺 |
|---|---|---|
| `hidden` | 完全神秘，不公開具體內容 | 只知道類型，例如「早餐福袋」 |
| `partial` | 半公開，描述可能內容 | 例如「可能含三明治、飯糰、吐司」 |
| `full` | 完全公開，列出主要內容 | 例如「含雞蛋三明治 1 份、鮪魚飯糰 1 份」 |

### status

| 狀態 | 說明 |
|---|---|
| `draft` | 草稿，尚未公開。 |
| `active` | 已上架，使用者可以預約。 |
| `sold_out` | 已售完或已被預約完。 |
| `expired` | 已過取貨時間。 |
| `hidden` | 商家手動隱藏或下架。 |

### quantity_available

第一版不一定要把 `quantity_available` 存成欄位，可以由後端計算：

```text
quantity_available = quantity_total - quantity_reserved
```

這樣可以避免資料不一致。

## reservations：預約紀錄

`reservations` 代表一般使用者預約某個福袋的紀錄。

因為 MVP 不做金流，所以不建議一開始叫 `orders`。  
`reservations` 比較接近實際流程：使用者先預約，到店出示取貨碼，商家確認取貨。

### 欄位

| 欄位 | 說明 | 範例 | 是否必填 |
|---|---|---|---|
| `id` | 預約 ID | `reservation_001` | 是 |
| `customer_id` | 一般使用者 ID，對應 `customers.id` | `customer_001` | 第二階段 |
| `merchant_id` | 預約哪間店，對應 `merchants.id` | `merchant_001` | 是 |
| `surprise_bag_id` | 預約哪個福袋，對應 `surprise_bags.id` | `bag_001` | 是 |
| `customer_name` | 使用者名稱 | `陳同學` | 否 |
| `customer_phone` | 使用者電話 | `0911222333` | 否 |
| `line_user_id` | LINE 使用者 ID，未來接 LIFF 時使用 | `Uxxxxxxxx` | 否 |
| `quantity` | 預約數量 | `1` | 是 |
| `pickup_code` | 取貨碼 | `4821` | 是 |
| `status` | 預約狀態 | `reserved`、`picked_up`、`cancelled`、`no_show` | 是 |
| `reserved_at` | 預約建立時間 | `2026-07-02T12:00:00+08:00` | 是 |
| `picked_up_at` | 完成取貨時間 | `2026-07-02T18:30:00+08:00` | 否 |
| `cancelled_at` | 取消時間 | `2026-07-02T13:00:00+08:00` | 否 |
| `customer_note` | 使用者備註 | `希望不要辣` | 否 |
| `merchant_note` | 商家內部備註 | `已電話確認` | 否 |

### status

| 狀態 | 說明 |
|---|---|
| `reserved` | 已預約，等待取貨。 |
| `picked_up` | 已完成取貨。 |
| `cancelled` | 使用者或商家取消。 |
| `no_show` | 使用者未取貨。 |

## customers：一般使用者

`customers` 代表一般使用者，也就是會從 LINE 或網頁進來瀏覽福袋、預約福袋、到店取貨的人。

MVP 可以先把使用者資訊直接存在 `reservations` 裡，例如 `customer_name`、`customer_phone`、`line_user_id`。

但第二階段建議補上 `customers`，因為之後會需要：

- 查看使用者的歷史預約。
- 統計 no-show 次數。
- 接 LINE LIFF 登入。
- 讓使用者取消預約或查看取貨碼。
- 做回訪、通知、會員資料管理。

### 欄位

| 欄位 | 說明 | 範例 | 是否必填 |
|---|---|---|---|
| `id` | 使用者 ID | `customer_001` | 是 |
| `display_name` | 顯示名稱 | `陳同學` | 否 |
| `phone` | 手機號碼 | `0911222333` | 否 |
| `email` | email | `customer@example.com` | 否 |
| `line_user_id` | LINE 使用者 ID，接 LIFF 後可取得 | `Uxxxxxxxx` | 否 |
| `status` | 使用者狀態 | `active`、`blocked` | 是 |
| `created_at` | 建立時間 | `2026-07-02T10:00:00+08:00` | 是 |
| `last_seen_at` | 最後使用時間 | `2026-07-02T12:00:00+08:00` | 否 |

### status

| 狀態 | 說明 |
|---|---|
| `active` | 正常使用者，可以預約。 |
| `blocked` | 被限制預約，例如惡意預約、長期 no-show。 |

### line_user_id

`line_user_id` 是 LINE 對每個使用者提供的唯一 ID。

之後接 LIFF 後，可以用它來辨識同一個使用者，讓使用者在 LINE 裡打開網頁時不需要重新輸入資料。

### 和 reservations 的關係

補上 `customers` 後，可以設計成：

```text
customers 1 - N reservations
```

也就是一個使用者可以有多筆預約。

### 為什麼 reservations 也存 merchant_id

理論上可以透過 `surprise_bag_id` 找到 `merchant_id`。

但 `reservations` 直接存 `merchant_id` 有幾個好處：

- 商家後台查詢自己的預約比較快。
- 之後就算福袋資料被修改，預約仍然知道當時是哪間店。
- API 實作比較簡單。

## 資料表關聯

### 關聯摘要

```text
merchants 1 - N merchant_users
merchants 1 - N surprise_bags
merchants 1 - N reservations
surprise_bags 1 - N reservations
customers 1 - N reservations
staff_users 1 - N approval_codes（第二階段，若建立 approval_codes）
```

### 詳細說明

| 關聯 | 說明 |
|---|---|
| `merchants.id` -> `merchant_users.merchant_id` | 一間店可以有多個管理者或員工帳號。 |
| `merchants.id` -> `surprise_bags.merchant_id` | 一間店可以上架多個福袋。 |
| `merchants.id` -> `reservations.merchant_id` | 一間店可以收到多筆預約。 |
| `surprise_bags.id` -> `reservations.surprise_bag_id` | 一個福袋可以被多個使用者預約，直到數量用完。 |
| `customers.id` -> `reservations.customer_id` | 一個使用者可以有多筆預約。第二階段補上 `customers` 後使用。 |
| `staff_users.id` -> `approval_codes.issued_by_staff_id` | 一位業務可以發出多組審核碼。第二階段補上 `approval_codes` 後使用。 |

## 審核碼的處理

目前你認為 `approval_codes` 暫時不需要，因為審核碼是一次性的。

MVP 可以先不建 `approval_codes` 表，先用較簡單的方式處理：

```text
商家註冊 -> 選擇 Google Maps 店家 -> 輸入審核碼 -> 通過後 merchants.status = approved
```

但要注意：如果系統真的要判斷「一次性」，長期來看還是需要某個地方記錄審核碼是否已被使用。

暫時可以有兩種做法：

| 做法 | 優點 | 缺點 |
|---|---|---|
| 固定審核碼或環境變數 | 最快，適合 demo | 不能真正做到一次性，也不知道誰用過 |
| 建 `approval_codes` 表 | 可以真正一次性、可追蹤業務發碼 | 多一張表與簡單後台 |

所以建議：

- Demo / 早期 MVP：先不做 `approval_codes` 表。
- 開始讓多位業務發碼時：再補 `approval_codes` 表。

## 其他暫緩的資料表

以下表不是第一版必做，但未來可能會需要。

| 表名 | 用途 | 何時需要 |
|---|---|---|
| `approval_codes` | 審核碼紀錄，例如誰發的、是否使用、綁到哪間店 | 需要真正一次性審核碼與追蹤發碼紀錄時 |

`staff_users` 和 `customers` 不是不做，而是第二階段補上；本文件已經先列出它們的欄位設計。

MVP 可以先把一般使用者資料直接存在 `reservations` 裡，例如 `customer_name`、`customer_phone`、`line_user_id`。等使用者功能變多，再正式啟用 `customers` 表，並讓 `reservations.customer_id` 指向 `customers.id`。

## MVP 第一版最重要的資料流

### 商家註冊

```text
1. 商家打開註冊頁
2. 商家在 Google Maps 搜尋並選擇店家
3. 系統取得 google_place_id、店名、地址、經緯度
4. 商家輸入審核碼
5. 審核碼正確後建立 merchants
6. 建立第一個 merchant_users，作為店主帳號
7. merchants.status = approved
```

### 商家上架福袋

```text
1. 商家登入
2. 建立 surprise_bags
3. 填寫福袋名稱、類型、售價、數量、取貨時間
4. 選擇 description_mode
5. status = active 後公開給使用者預約
```

### 使用者預約福袋

```text
1. 使用者瀏覽 active 的 surprise_bags
2. 選擇福袋並送出預約
3. 系統建立 reservations
4. 系統產生 pickup_code
5. surprise_bags.quantity_reserved 增加
6. 使用者到店出示 pickup_code
7. 商家確認後 reservations.status = picked_up
```
